# 用户画像算法设计

这份设计对应当前项目里的 `backend/core/profile.js`。它不是照搬某篇论文的重模型，而是把推荐系统和对话画像里的几类稳定方法工程化：隐式反馈、时间衰减、规则抽取、短期会话信号、人工确认优先。

## 目标

用户不填写表单时，系统也能从以下行为里自动更新画像：

- 听了什么歌、歌手、榜单、语种。
- 在什么天气、什么时段听了什么歌。
- 点歌时说了什么风格、场景和要求。
- 聊天时表达了哪些偏好，比如“这首歌背景是什么”“别突然切歌”“TTS 听不清”。
- 切歌、自动下一首、定时播放带来的实际播放行为。

画像最终服务三件事：

- 让 GPT 推荐时更贴近用户口味。
- 让 DJ 口播更像用户想听的电台风格。
- 让前端 SETTINGS 里可以把自动建议确认进手动画像。

## 数据结构

画像分两层：

```json
{
  "taste": {},
  "scenes": {},
  "djStyle": {},
  "notes": "",
  "auto": {
    "version": 1,
    "updatedAt": "",
    "summary": "",
    "stats": {
      "playEvents": 0,
      "chatEvents": 0,
      "toplistEvents": 0
    },
    "evidence": []
  }
}
```

手动层是 `taste/scenes/djStyle/notes`，优先级最高。自动层是 `auto.evidence`，只由系统维护。

每条证据：

```json
{
  "kind": "genre",
  "value": "电音",
  "weight": 2.2,
  "source": "toplist",
  "at": "2026-05-13T12:00:00.000Z",
  "note": "换几首电音榜的"
}
```

字段含义：

- `kind`：画像维度，比如 `artist`、`genre`、`language`、`tempo`、`avoid`、`scene.work`、`dj.songIntro`。
- `value`：具体偏好值。
- `weight`：证据强度。
- `source`：来自 `chat`、`direct`、`toplist`、`next`、`scheduler` 等。
- `at`：发生时间。
- `note`：用于解释的短文本。

上下文类证据会使用前缀命名，例如：

```json
{ "kind": "context.weather.rain.song", "value": "Tycho - Hours", "weight": 1.2 }
{ "kind": "context.time.lateNight.song", "value": "Bonobo - No Reason", "weight": 1.1 }
```

## 输入信号

### 播放信号

触发函数：`learnProfileFromPlay(song, context)`

输入：

- 当前真正开始播放的歌曲。
- 点歌原文。
- 榜单信息。
- 播放来源。

提取内容：

- 歌手：`artist`
- 语种：根据歌名和歌手名推断中文、英文、日语、韩语。
- 榜单曲风：比如电音榜、国风榜、说唱榜。
- 点歌文本里的场景和偏好词。
- 天气和时间段上下文：雨天、晴天、深夜、早晨等。

播放行为是强隐式反馈，所以权重较高。例如听到 Tycho，会写入：

```json
{ "kind": "artist", "value": "Tycho", "weight": 1.4 }
```

如果来自电音榜，会写入：

```json
{ "kind": "genre", "value": "电音", "weight": 2.2 }
{ "kind": "tempo", "value": "节奏感强", "weight": 1.5 }
```

如果在雨天播放了一首歌，会写入：

```json
{ "kind": "context.weather.rain.song", "value": "Tycho - Hours", "weight": 1.2 }
{ "kind": "context.weather.rain.artist", "value": "Tycho", "weight": 0.9 }
```

如果在深夜播放了一首歌，会写入：

```json
{ "kind": "context.time.lateNight.song", "value": "Bonobo - No Reason", "weight": 1.1 }
{ "kind": "context.time.lateNight.artist", "value": "Bonobo", "weight": 0.8 }
```

### 对话信号

触发函数：`learnProfileFromConversation(userInput, assistantOutput, context)`

输入：

- 用户原话。
- AI 回复。
- 当前播放歌曲。
- 榜单上下文。

提取内容：

- 风格偏好：电音、国风、民谣、R&B、City Pop 等。
- 场景偏好：工作、深夜、通勤、睡前、雨天。
- 避免项：太吵、太甜、太机械、别突然切歌。
- DJ 风格：喜欢背景故事、创作状态、自然过渡、像真人聊天。
- 音频偏好：TTS 音量、人声清晰、音乐压低比例、语速。

例子：

用户说“这首歌是在什么状态下写的，有什么背景故事吗”，写入：

```json
{ "kind": "dj.songIntro", "value": "喜欢歌曲背景、创作状态和作品位置", "weight": 3.74 }
```

用户说“别突然切歌，我想先聊聊”，写入：

```json
{ "kind": "dj.tone", "value": "聊天时先保持当前播放，不要主动切歌", "weight": 3.74 }
```

## 证据更新

当前实现采用 append-only evidence，不直接覆盖旧画像。

流程：

1. 从播放或对话中抽取证据。
2. 去重同一次事件里的重复证据。
3. 追加到 `auto.evidence`。
4. 超过上限时只保留最近 `160` 条。
5. 重新生成 `auto.summary`。

伪代码：

```text
for each event:
  signals = extract(event)
  signals = unique(signals by kind + value + source)
  profile.auto.evidence.append(signals)
  profile.auto.evidence = last_160(profile.auto.evidence)
  profile.auto.summary = summarize(decayed_scores(profile.auto.evidence))
```

## 时间衰减

画像不是简单计数，而是按时间衰减。越新的行为影响越大，旧偏好会慢慢淡出。

当前半衰期：`21` 天。

公式：

```text
score(kind, value) = Σ weight_i * 0.5 ^ (age_i / half_life)
```

其中：

- `weight_i` 是证据权重。
- `age_i` 是这条证据距离现在的时间。
- `half_life` 是 21 天。

含义：

- 今天表达过的偏好权重完整保留。
- 21 天前的同等证据只剩一半影响。
- 多次重复出现的偏好会自然叠加。

## 聚合输出

系统按 `kind` 聚合分数，然后取 Top 值。

映射关系：

| kind | 输出字段 |
|---|---|
| `artist` | `auto.taste.favoriteArtists` |
| `genre` | `auto.taste.favoriteGenres` |
| `language` | `auto.taste.favoriteLanguages` |
| `tempo` | `auto.taste.preferredTempo` |
| `avoid` | `auto.taste.avoid` |
| `scene.work` | `auto.scenes.work` |
| `scene.lateNight` | `auto.scenes.lateNight` |
| `scene.commute` | `auto.scenes.commute` |
| `scene.sleep` | `auto.scenes.sleep` |
| `dj.tone` | `auto.djStyle.tone` |
| `dj.songIntro` | `auto.djStyle.songIntro` |
| `dj.transition` | `auto.djStyle.transition` |
| `audio.balance` | `auto.audio` |

上下文证据按前缀聚合到：

- `auto.context.weather`
- `auto.context.time`

然后拼成自然语言摘要，例如：

```text
偏好电音、国风/古风；常听 Tycho、Bonobo；
节奏倾向是节奏感强；歌曲介绍上喜欢背景、创作状态和作品位置；
过渡上切歌时提前铺垫，自然过渡，别硬切。
```

## 冲突处理

优先级：

```text
手动画像 > 明确避免项 > 最近强信号 > 长期弱信号
```

规则：

- 用户手动保存的 `taste/scenes/djStyle` 不被自动层直接覆盖。
- 自动层只作为上下文补充，或者在 SETTINGS 的 LEARN 里生成候选建议。
- 出现“不要/别/太吵/太甜”这类表达时，写入 `avoid`，比普通喜欢信号更谨慎。
- 如果用户在聊天模式里提问，画像学习可以记录偏好，但不能触发换歌。
- 播放行为只说明“可能喜欢”，不等于绝对喜欢，所以用时间衰减和多次累积确认。

## 推荐链路

当前使用方式：

```text
播放/聊天事件
  -> learnProfileFromPlay / learnProfileFromConversation
  -> auto.evidence
  -> deriveAutoProfile
  -> formatProfileForPrompt
  -> GPT-5.5 选歌和口播
```

`formatProfileForPrompt` 会把手动层和自动层一起塞进 system prompt。GPT 看到的是：

- 手动音乐口味。
- 手动场景偏好。
- DJ 风格。
- 系统自动学习画像。

这样既保留可控性，又能让系统随着使用变聪明。

## 为什么不用重模型

BERT4Rec、SASRec、GRU4Rec 这类模型适合大量用户和大量交互数据。这个项目目前是个人电台，数据量很小，直接上深度模型会有几个问题：

- 冷启动阶段没有足够训练样本。
- 训练和部署成本不值得。
- 可解释性差，不方便用户修正。
- 当前需求更需要“实时、可解释、可编辑”的画像。

所以当前实现采用轻量在线画像算法。它更像推荐系统里的 baseline，但很适合个人助理场景。

## 后续升级

当播放历史足够多时，可以加三层增强：

1. 短期会话向量：最近 5 到 10 首歌单独建短期偏好，用于“这一小时想听什么”。
2. 歌曲 embedding：把歌曲的风格、语速、情绪、年代向量化，避免只靠歌手和文本规则。
3. 序列模型：有足够数据后再接 SASRec/BERT4Rec，用最近播放序列预测下一首候选。

升级时仍建议保留当前 `auto.evidence`，因为它可解释、可回滚、可展示给用户确认。
